# KBO 기록실 크롤러 및 Vue3 시각화 계획

## 목표

- KBO `기록・순위 > 기록실`에 있는 선수 기록, 팀 기록, 투수 VS 타자 데이터를 수집한다.
- 수집한 데이터를 MySQL `kbo` 데이터베이스에 적재한다.
- 적재된 데이터를 Vue3 웹 화면에서 테이블 형태로 조회하고 시각화한다.
- Python, SQLModel, Vue3를 중심으로 최소 기능부터 동작하게 만든다.

## 수집 범위

1차 구현은 기록실의 정규 테이블 데이터를 대상으로 한다.

- 선수 기록: 타자, 투수, 수비, 주루
- 팀 기록: 타자, 투수, 수비, 주루
- 투수 VS 타자: 선수 선택 흐름을 확인한 뒤 수집 범위를 확정

아래 항목은 기록실과 인접하지만 1차 범위에서 제외한다.

- 팀 순위
- 선수 순위
- 역대 기록
- 예상 달성 기록
- 기록 정정 현황
- 관중 현황
- 퓨처스리그 기록실

필요하면 위 항목은 별도 2차 범위로 계획한다.

## 현재 전제

- DB: MySQL
- DB 위치: WSL 로컬 `localhost:3306`
- DB 이름: `kbo`
- DB 사용자: `jys`
- DB 비밀번호는 `.env`에만 저장한다.
- 백엔드와 프론트엔드는 같은 저장소 안에서 관리한다.

## 성공 기준

- 크롤러가 기록실 카테고리, 페이지 종류, 조회 조건을 메타데이터로 관리한다.
- 각 기록 테이블의 행과 컬럼을 원본 구조 그대로 수집한다.
- 수집 데이터가 SQLModel 모델을 통해 MySQL에 저장된다.
- 동일 조건을 반복 수집해도 의도치 않은 중복이 생기지 않는다.
- API를 통해 카테고리별 기록 목록을 조회할 수 있다.
- Vue3 화면에서 카테고리와 조회 조건을 선택하고 결과 테이블을 볼 수 있다.
- `.env`만 수정하면 DB 접속 정보를 변경할 수 있다.

## 기술 방향

- 크롤링
  - 기본은 `requests` 기반 HTML 파싱으로 시작한다.
  - ASP.NET WebForms postback, 동적 select, 페이지 이동 처리가 필요한 화면은 `playwright`를 사용한다.
  - 구현 전에 각 화면의 form field, hidden field, pagination 방식을 샘플로 검증한다.
- DB 모델
  - SQLModel 사용
  - 기록 종류별 고정 컬럼 테이블을 우선 검토한다.
  - 컬럼 차이가 큰 상황별 기록은 `record_type`, `condition`, `columns`, `values`를 보존하는 원본형 테이블도 검토한다.
- API 서버
  - Python 기반 경량 API
  - Vue3 테이블 조회에 필요한 카테고리 목록, 조건 목록, 기록 조회 API를 먼저 만든다.
- 프론트엔드
  - Vue3
  - 기록 카테고리 선택, 조건 선택, 테이블 조회를 첫 화면으로 만든다.

## 환경 변수 계획

`.env`에는 아래 값을 둔다.

```env
DB_HOST=localhost
DB_PORT=3306
DB_NAME=kbo
DB_USER=jys
DB_PASSWORD=<local-password>
```

필요하면 구현 단계에서 `DATABASE_URL`을 조립해서 사용한다.
